**Structure:**

1. **Title + badges**: `visa-bot` -- Multi-tenant US visa appointment monitor and auto-rescheduler
2. **What it does** (2-3 sentences): Monitors `ais.usvisa-info.com` for B1/B2 visa appointment availability. When a better date appears, it automatically reschedules. Supports multiple users/bots with independent configurations.
3. **Key features** (bullet list):
   - Multi-tenant: multiple bots with independent credentials, schedules, and configurations
   - Auto-reschedule: detects earlier dates and reschedules automatically with safety guards
   - Smart polling: configurable intervals per locale, super-critical burst mode, TCP backoff
   - Date failure tracking: stops wasting polls on repeatedly-failing dates (sliding window + CAS escape hatch)
   - Proxy support: direct, Webshare, Bright Data, Firecrawl providers with circuit breaker
   - Dashboard: web UI for monitoring bot status, poll logs, reschedule history
   - Notifications: email (Resend) + webhook (HMAC-signed) on reschedule events
   - Encrypted credentials: AES-256-GCM at rest
   - Dual deployment: cloud (Trigger.dev) + local (RPi with residential IP)
4. **Architecture overview** (brief):
   - Stack: Node.js ESM/TS, Hono API, Trigger.dev v4, Neon PostgreSQL + Drizzle ORM
   - Core flow: `poll-cron` -> `poll-visa` (fetch dates -> reschedule inline -> self-trigger chain)
   - Login: pure fetch (no browser/Playwright needed), ~1s
   - API: Hono REST endpoints for bot management, logs, health
5. **Supported locales**: es-co (Colombia), es-pe (Peru) -- extensible to other US embassy locales
6. **Prerequisites**: Node.js 20+, PostgreSQL (Neon recommended), Trigger.dev v4 account
7. **Setup**:
   - Clone repo
   - `npm install`
   - Copy `.env.example` to `.env` and fill in values (reference .env.example)
   - `npm run db:push` to create tables
   - `npm run dev` for API server
   - `npm run trigger:dev` for background worker
   - Create a bot via `POST /api/bots` with credentials
8. **API endpoints** (table format -- from CLAUDE.md, but generalized):
   - GET/PUT /api/bots/:id, POST activate/pause/resume, DELETE
   - POST /api/bots/validate-credentials
   - GET /api/bots/:id/logs/polls, reschedules, cas-prefetch
   - GET /api/health
9. **Configuration** (per-bot):
   - `targetDateBefore`: only reschedule to dates before this
   - `maxReschedules` / `rescheduleCount`: limit total reschedules
   - `excluded_dates`: date ranges to skip
   - `pollEnvironments`: where this bot polls (dev/prod/both)
   - `proxyProvider`: direct, webshare, brightdata, firecrawl
10. **Safety guards**:
    - Never reschedules to equal or later date (original slot lost permanently)
    - `maxReschedules` hard limit
    - Excluded date ranges respected
    - Dry-run by default for manual scripts
    - Peru (es-pe): 2 reschedule limit, irreversible block
11. **Deployment**:
    - Cloud: Trigger.dev deploy
    - RPi: `npm run deploy:rpi` (residential IP for lower block risk)
12. **Scripts** (only the core ones from package.json):
    - `npm run login -- --bot-id=N` -- manual login
    - `npm run test-reschedule -- --bot-id=N` -- dry-run reschedule test
    - `npm run monitor` -- live monitoring
    - `npm test` -- run test suite
